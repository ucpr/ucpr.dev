# Blog Review 

このコマンドでは、ブログ記事がテクニカルライティングの観点から適切かどうかをレビューします。以下のレビューフローに従って、記事の内容、構成、明確さ、一貫性、正確性、文法などを評価し、必要に応じて改善点を指摘します。

特に以下を意識してレビューを行います。

- **文書から得られるアウトプットが明確で、読み手のレベルによらず一定に伝わること**

## レビューフロー

0. **準備**: レビュー対象のブログ記事がどれかをユーザーに **必ず** 確認します。
1. **記事の内容確認**: 指定されたブログ記事の内容を最初から最後まで通読し、主題や目的を理解します。
2. **構成の評価**: 記事の構成が論理的であるか、段落やセクションが適切に分けられているかを確認します。
3. **明確さのチェック**: 専門用語や技術的な概念が明確に説明されているか、読者が理解しやすいかを評価します。
4. **一貫性の確認**: 用語の使用やスタイルが一貫しているかをチェックします。
5. **正確性の検証**: 技術的な内容が正確であり、最新の情報に基づいているかを確認します。
6. **文法とスペルのチェック**: 文法的な誤りやスペルミスがないかを確認します。
7. **フィードバックの提供**: 改善点や良い点を具体的に指摘し、建設的なフィードバックを提供します。
8. **最終確認**: 修正後の記事を再度確認し、すべての問題が解決されているかを確認します。
9. **承認または再レビュー**: 記事が基準を満たしている場合は承認し、必要に応じて再レビューを行います。

### Technical Writing One Point

- 用語は文書内で一貫性をもたせる
- 曖昧な代名詞の利用は避ける
- 受動態は避け、能動態を優先する
- 強い動詞を選択する1
- 曖昧な動詞よりも意味が限定される動詞を選ぶ
- 1文1メッセージ
- 長い文章はリスト形式に変換する
- 不要な言葉は排除する
- 順序が重要な場合は番号付きリストを使用し、順序が重要でない場合は箇条書きリストを使用する
- リストを使う場合は言葉の並列性を保つ
- 番号の付いたリストは命令語で始める
- リストと表を適切に使う
- 段落の骨子を示す良いリード文を作る
- 1段落1トピックに絞る
- 読み手に伝えるべきことを定義する
- 読み手を意識して書く
- ドキュメントの最初にそのドキュメントの重要なポイントを明確に示す

### Technical Writing Two Point

- ドキュメントの対象範囲と前提条件を最初に示す
- できるだけタスクベースの見出しにする
- （一気にまとめて出すのではなく）情報（新しい概念や指示）は小出しにする
- 1つの図面内の情報量を制限する
- キャプションで要点を説明するか、画像に注意を惹くマークを追加して、画像または図の適切な箇所に読者の注意を向ける
- 理解しやすい簡潔なサンプルコードを作成する
- コードのコメントは短くするが、簡潔さよりも明快さを優先する
- 自明なコードにコメントは書かない
- コメントは、コード内の直感的でない部分を明らかにすることに集中させる
- 良い例だけでなく、良くない例も提供する
- コードサンプルは複数の複雑さを想定したものを提供する

## レビューに際しての注意点

- ユーザーにレビュー対象のブログ記事を必ず確認すること。
- レビューは建設的かつ具体的に行い、改善点を明確に指摘すること。
- 技術的な正確性を最優先に考慮すること。
- レビュー結果をユーザーに分かりやすく伝えること。
- 必要に応じて、追加のリソースや参考資料を提供すること。
- レビューの過程で発見した良い点も積極的にフィードバックすること。
- ユーザーの意図や目的を尊重し、適切なアドバイスを提供すること。
- レビュー後のフォローアップを行い、必要に応じて再レビューを実施すること。
- レビューの際には、最新のテクニカルライティングのベストプラクティスを参考にすること。
- ユーザーが改善点を実践できるよう、具体的なアクションプランを提案すること。
